hedonistic-pki

PKI ceremonies are miserable. You spend a week reading OpenSSL man pages, writing shell scripts held together with prayer, and praying you typed the right flags on your airgapped laptop. This tool exists so you never have to do that again.

Write a YAML file that describes your certificate hierarchy. Run one command. Walk away with encrypted keys, signed certificates, post-quantum key bundles, paper backups, and deployment archives. The entire thing is a single static binary with no runtime dependencies. Not even OpenSSL.

Download v1.0.0 Source on GitHub

The problem

Every organization that takes security seriously eventually needs its own PKI. A root CA that lives in a safe. An intermediate that signs things. Leaf certs for code signing or mutual TLS. Maybe post-quantum keys if you're thinking about the future.

The tooling for this is terrible. You either pay a vendor tens of thousands of dollars, or you cobble together a pile of openssl req and openssl ca commands and hope the extensions are right. You Google "openssl create intermediate CA" for the fourth time this year. You wonder if you got the pathlen constraint backwards. You definitely forgot to set the key usage bits.

hedonistic-pki replaces all of that with a config file: 

name: my-company-pki
organization: "My Company"

hierarchy:
  - name: root-ca
    cn: "My Company Root CA"
    cert_type: root
    algorithm: rsa_4096
    validity:
      years: 20
    offline: true

  - name: intermediate
    cn: "My Company Intermediate CA"
    cert_type: intermediate
    parent: root-ca
    algorithm: rsa_4096
    validity:
      years: 10
    pathlen: 0

  - name: code-signing
    cn: "My Company Code Signing"
    cert_type: leaf
    parent: intermediate
    algorithm: rsa_4096
    validity:
      years: 2
    extensions:
      extended_key_usage: [codeSigning]

That is your entire ceremony. Run hedonistic-pki ceremony --config ceremony.yaml and it generates everything: the root CA, the intermediate signed by the root, the code signing cert signed by the intermediate, post-quantum keys in parallel, paper backups for your safe, and deployment archives sorted by classification.

What you get

One binary, no dependencies. It compiles to a static binary around 2MB. There is nothing to install, no shared libraries to worry about, no OpenSSL version conflicts. Copy it to a USB stick and run it on your airgapped ceremony machine.

Config-driven hierarchies. Describe your certs as a flat YAML list with parent references. The tool topologically sorts them and generates certificates in the right order. You can have as many tiers as you want: root, intermediate, sub-CA, leaf.

Post-quantum cryptography. Every cert can optionally generate ML-DSA-87 signing keys (NIST FIPS 204) and ML-KEM-1024 encapsulation keys (NIST FIPS 203) alongside classical RSA-4096 or Ed25519. This is a hybrid "and" model: both chains must verify. Set algorithm: ml_dsa_87 on any cert, or use parallel_keys to generate both classical and PQ keys.

Memory hardening. Private keys live in a ChaCha20-Poly1305 encrypted vault in memory for the entire ceremony. The vault key comes from the OS CSPRNG and never touches disk. On Linux, memory is mlock'd and core dumps are disabled. All sensitive buffers are zeroized on drop.

Encrypted disk output. Private keys hit disk encrypted with PKCS#8 + AES-256-CBC. You provide passphrases interactively. Nothing is written in plaintext.

Paper backups. The tool generates a printable HTML file with QR codes encoding your private keys. Large RSA keys get split across multiple QR codes. Each page includes a SHA-256 fingerprint for verification after scanning. Print it, laminate it, put it in a fireproof safe.

Deployment packages. After the ceremony, outputs are automatically classified: offline keys that belong in a vault, deploy keys for your CI system, public certs for distribution. Each classification gets its own tar.gz archive with a JSON manifest.

Passphrase vault. During a ceremony you type a lot of passphrases. The tool can encrypt them all into a single AES-256-GCM vault file. Take it to a networked machine later, run vault-decrypt, and transfer everything to 1Password.

Airgap by design. No network calls. No telemetry. No update checks. No DNS lookups. Nothing. The binary embeds an encrypted copy of its own source code so you can recompile it during key rotation without trusting anything off the machine.

Getting started

Install

Grab a pre-built binary from the latest release:

# Pick your platform
curl -LO https://github.com/hedonistic-io/hedonistic-pki/releases/latest/download/hedonistic-pki-linux-x86_64
curl -LO https://github.com/hedonistic-io/hedonistic-pki/releases/latest/download/hedonistic-pki-linux-arm64
curl -LO https://github.com/hedonistic-io/hedonistic-pki/releases/latest/download/hedonistic-pki-darwin-arm64
curl -LO https://github.com/hedonistic-io/hedonistic-pki/releases/latest/download/hedonistic-pki-darwin-x86_64

chmod +x hedonistic-pki-*

# Verify the download
curl -LO https://github.com/hedonistic-io/hedonistic-pki/releases/latest/download/hedonistic-pki-linux-x86_64.sha512
shasum -a 512 -c hedonistic-pki-linux-x86_64.sha512

Or build from source (requires Rust 1.93+):

git clone https://github.com/hedonistic-io/hedonistic-pki.git
cd hedonistic-pki
cargo build --release
# binary is at target/release/hedonistic-pki

Your first ceremony

Start with a dry run. Write a ceremony config (or use the quickstart example) and validate it: 

hedonistic-pki ceremony --config ceremony.yaml --dry-run

The dry run parses your config, validates parent references, checks for cycles, and prints the generation plan without creating any keys. When it looks right: 

hedonistic-pki ceremony --config ceremony.yaml --output /mnt/usb/pki

You will be asked for passphrases. Certs that share a parent automatically share a passphrase, so you type fewer passwords than you have certs. Certs marked no_passphrase: true skip the prompt entirely.

When it finishes, you have a directory full of encrypted keys, signed certificates, post-quantum key bundles, a paper backup HTML file, and deployment archives. Copy the offline keys to cold storage. Deploy the rest.

Commands

The main event. Reads your config and runs the whole ceremony.

generate

Quick three-tier PKI without a config file. Generates a root CA, intermediate CA, and code signing leaf with sensible defaults. Good for testing or simple setups. Use ceremony for anything more complex.

FlagDefaultDescription
--output, -orequiredOutput directory
--min-passphrase-length16Minimum passphrase length
hedonistic-pki generate --output /mnt/usb/pki

This produces the following directory structure:

/mnt/usb/pki/ root-ca/ root-ca.crt 20-year RSA-4096 root certificate root-ca.key PKCS#8 encrypted private key intermediate-ca/ intermediate-ca.crt 10-year intermediate certificate intermediate-ca.key PKCS#8 encrypted private key chain.crt full chain (intermediate + root) code-signing/ code-signing.crt 2-year code signing certificate code-signing.key PKCS#8 encrypted private key pq/ root-ca.vk / .sk ML-DSA-87 verification/signing keys intermediate-ca.vk / .sk ML-DSA-87 verification/signing keys code-signing.vk / .sk ML-DSA-87 verification/signing keys code-signing.ek / .dk ML-KEM-1024 encapsulation/decapsulation keys manifest.json PQ key manifest with hex-encoded public keys *.endorsement.sig PQ cross-signing chain

rekey

Rotate your PKI. Generates a fresh chain, creates a CRL revoking the old intermediate, and extracts the embedded source so you can recompile the binary with the new CA. You need the old root CA passphrase (for the CRL) and three new passphrases.

FlagDefaultDescription
--old-pkirequiredExisting PKI directory to revoke
--output, -orequiredOutput directory for the new chain
--min-passphrase-length16Minimum passphrase length for new keys
hedonistic-pki rekey --old-pki /mnt/usb/pki --output /mnt/usb/pki-v2

paper-backup

Generate a paper backup from an existing PKI directory. Scans for private keys and produces a printable HTML file with QR-encoded key material.

FlagDefaultDescription
--pki-dir, -prequiredPKI directory to scan for keys
--output, -oautoOutput HTML file path
hedonistic-pki paper-backup --pki-dir /mnt/usb/pki

vault-decrypt

Decrypt a ceremony passphrase vault. After a ceremony, you might have saved all your passphrases into an encrypted vault file. Take that file to a networked machine and use this command to get them into your password manager.

Without --ramfs, passphrases are printed to the terminal and a disk-persistence warning is displayed (terminal scrollback, shell history, and screen recording can all capture the output). You must confirm to proceed.

With --ramfs, passphrases are written to a RAM-backed filesystem that never touches persistent storage. On Linux this uses /dev/shm (no root required). On macOS it creates a RAM disk via hdiutil. A background process automatically overwrites and unmounts the filesystem after --ttl seconds.

FlagDefaultDescription
--vaultrequiredPath to encrypted vault file
--ramfsDecrypt to a temporary RAM filesystem instead of terminal
--ttl300Auto-destruct timeout in seconds (0 to disable)
# Recommended: decrypt to RAM filesystem (self-destructs in 5 minutes)
hedonistic-pki vault-decrypt --vault ./pki/ceremony-vault.enc --ramfs

# Custom timeout (10 minutes)
hedonistic-pki vault-decrypt --vault ./pki/ceremony-vault.enc --ramfs --ttl 600

# Legacy: print to terminal (shows disk-persistence warning)
hedonistic-pki vault-decrypt --vault ./pki/ceremony-vault.enc
# after transferring to 1Password:
shred -u ceremony-vault.enc

extract-source

Pull the encrypted source code out of the binary. The source is embedded at compile time for self-recompilation during rekey.

FlagDefaultDescription
--output, -orequiredDirectory for the extracted source archive
--keyrequiredCode signing private key for decryption
hedonistic-pki extract-source --output ./source --key ./pki/code-signing/code-signing.key

verify

Print build information and instructions for verifying the binary's CMS signature. No flags. 

hedonistic-pki verify

Post-quantum selection

There is no top-level PQ config block. You choose algorithms per cert. To get both classical and PQ keys for the same cert, use parallel_keys: 

- name: hybrid-root
  cn: "Hybrid Root CA"
  cert_type: root
  algorithm: rsa_4096
  validity:
    years: 20
  parallel_keys:
    - ed25519
    - ml_dsa_87

Passphrase grouping

Passphrases are grouped automatically. There is no config field for it. The rules are simple: each root CA gets its own passphrase. Non-root certs that share the same parent share a passphrase. Certs with no_passphrase: true get nothing.

How it protects your keys

In memory

Every private key generated during a ceremony is immediately encrypted with ChaCha20-Poly1305 using a 256-bit key from the OS CSPRNG. That vault key exists only in process memory and is never serialized. When a key needs to sign a child certificate, it is decrypted into a Zeroizing<Vec<u8>> buffer, used, and wiped. Plaintext private keys do not persist in unprotected memory at any point during the ceremony.

On Linux

At startup, the binary calls mlockall to prevent the kernel from swapping any memory pages to disk. It sets PR_SET_DUMPABLE to zero so core dumps cannot leak key material. The release binary is compiled with panic=abort to prevent stack unwinding from exposing data.

On macOS and other platforms, the vault encryption and zeroization still apply but the OS-level hardening is not available.

On disk

Private keys are written encrypted with PKCS#8 + AES-256-CBC. The passphrase you provide during the ceremony is the decryption key. If OpenSSL is not available (which is fine, the tool does not require it), keys are written with restrictive file permissions as a fallback.

Post-quantum

The tool generates post-quantum keys using the NIST-standardized algorithms from FIPS 203 and FIPS 204. ML-DSA-87 (formerly Dilithium) provides digital signatures at security level 5. ML-KEM-1024 (formerly Kyber) provides key encapsulation at the same level. The classical X.509 chain and the PQ endorsement chain are independent but cross-signed. Verification requires both to pass.

Embedded source

The binary contains an encrypted copy of its own source code, signed with the code signing certificate. During a rekey ceremony, it extracts this source and writes a recompilation script so you can rebuild the binary with the new CA without fetching anything from the network.

Paper backups

Paper backup HTML files contain one page per key. Each page has the certificate name, generation timestamp, SHA-256 fingerprint, a QR code (or multiple QR codes for large RSA keys), and the raw PEM text as a fallback. To reconstruct: scan all QR parts for a key, concatenate them, verify the SHA-256 checksum printed on the page, and you have the PEM-encoded private key.

Building from source

You need Rust 1.93 or later. The repo pins the toolchain in rust-toolchain.toml.

cargo build            # debug build
cargo build --release  # optimized, ~2MB binary
cargo test             # 99 tests

The release profile uses opt-level = "z" (size optimization), LTO, single codegen unit, stripped symbols, and panic=abort.

Cross-compilation works with standard Rust targets. There is a build-linux.sh script for building Linux binaries from macOS.

The build.rs script handles embedding the encrypted source archive. It compresses src/ at compile time so the binary is self-contained for rekey operations.
